# 产品发布节奏规划

**原则**：优化用户价值 × 技术依赖 × 反馈周期 × 个人精力

---

## 1. 产品生命周期分析

### 1.1 关键指标

**用户价值** (完整度 0-100%):
- 0% = 无法用
- 30% = 最小可用（只能计时）
- 60% = 有用（计时 + 待办）
- 80% = 很好用（+ 分析）
- 100% = 完整（所有功能）

**技术复杂度**:
- 简单 (1 周): 基础 UI、简单数据库操作
- 中等 (2-3 周): 状态管理、列表交互
- 复杂 (3+ 周): 图表、同步、小组件

**用户反馈价值**:
- 高: 核心流程（计时、待办）
- 中: 增强体验（通知、分析）
- 低: 可选功能（标签、导出）

---

## 2. 三层发布策略

### 2.1 Alpha (V0.1 ~ V0.3) — 内部验证

**目标**: 验证核心计时流程可用，不对外发布

**功能范围** (30% 完整度):
- [x] 计时器（番茄钟 + 自由计时）
- [x] 基础 UI（开始/暂停/停止）
- [x] 本地数据库存储
- [x] 简单通知

**时间**: 4 周

**发布方式**: 内部测试（自用 + 邀请朋友）

**关键验证点**:
- ✅ 计时精度是否准确？
- ✅ 计时记录是否稳定保存？
- ✅ 日常使用是否流畅？

**成果物**:
```
- v0.1: 可以启动计时
- v0.2: 可以暂停/继续/停止，数据持久化
- v0.3: 添加倒计时，验证7种计时模式都可用
```

**收获**:
- 发现计时引擎的 Bug
- 用户是否真的用计时？还是只在试？
- UI 流畅度反馈

**时间轴**:
```
W1-W2: 搭 Drift 数据库 + 基础计时页面
  └─ v0.1 可运行

W2-W3: 状态管理（Riverpod）+ 数据持久化
  └─ v0.2 计时可保存

W3-W4: 添加其他计时模式 + 通知
  └─ v0.3 完整计时体验
```


## 3. 最优发布节奏

### 3.1 时间表 (推荐方案)

```
┌─ Alpha (内部) ─────────────────────┐
│                                     │
│ W1-W4: 核心计时 (v0.1 ~ v0.3)      │ (自用 + 邀请 3-5 人测)
│                                     │
└─────────────────────────────────────┘
          ↓
┌─ Beta (开源社区) ────────────────────┐
│                                      │
│ W5-W6: 待办列表 (v0.4)              │ (GitHub Release)
│        用户反馈收集中                │
│                                      │
│ W7-W8: 热力图分析 (v0.5 ~ v0.6)    │ (正式发布 v1.0.0)
│        功能迭代                      │
│                                      │
│ W9-W10: 高级功能 (v0.7 ~ v0.8)     │ (待办优先级/标签)
│         选做小组件或跳过             │
│                                      │
└──────────────────────────────────────┘
          ↓
┌─ Stable (应用商店) ─────────────────┐
│                                     │
│ W11-W13: 性能优化 + 文档 (v1.0)     │ (应用市场上线)

```

**关键时间点**:
- **W4 末**: 验证计时是否能用
- **W6 末**: 验证待办 + 计时的组合是否足够用
- **W8 末**: 版本 v1.0.0 正式发布
- **W13 末**: 应用商店发布

**总计**: 13 周 (3+ 个月)

---

### 3.2 关键决策

#### 决策 1: 小组件要不要在 Beta 中做？

| 方案 | 好处 | 代价 |
|------|------|------|
| **Beta 中做（W9-W10）** | 完整体验，用户留存率高 | 延长 2 周，个人精力大 |
| **V1.1 再做** | 快速上线，快速反馈 | 用户体验不完整 |
| **永不做** | 专注核心，代码简洁 | 失去一个差异化点 |

**建议**: **V1.0 先不做，等发布后根据用户反馈再决定** ← 最符合产品节奏

---

#### 决策 2: 什么时候对外公开宣传？

| 时间 | 方式 | 效果 |
|------|------|------|
| **W4 (v0.3)** | 技术博客分享逆向分析 | 吸引开源爱好者 |
| **W8 (v1.0)** | Product Hunt / GitHub Trending | 获得最大关注 |
| **W13 (应用商店)** | App Store 上线 | 吸引普通用户 |

**建议**: 
- W4 分享"我怎么用 Flutter 复刻 Ticking"（技术向）
- W8 宣传 v1.0 发布（功能完整）
- W13 应用商店发布（渠道拓展）

---

#### 决策 3: 怎么处理用户反馈？

```
v0.4 (W5-W6)
  ↓
"用户反馈：想要任务优先级"
  ↓
v0.5 (W7) 加入优先级功能
  ↓
"用户反馈：热力图太简单"
  ↓
v0.6 (W8) 升级热力图
```

**原则**:
- Alpha: 忽略反馈（验证自己的想法）
- Beta: 快速反应（1-2 周内修复 Bug，1 周内添加高频需求）
- Stable: 有节制地反应（计划版本中再加）

## 5. 另一个角度：竞争优势时间表

### 5.1 何时建立竞争优势？

```
计时器 (纯 Flutter)
  ↓ W4
计时 + 待办 (对标 Ticking)
  ↓ W8
计时 + 待办 + 热力图 (功能完整)
  ↓ W9 (选项 A)
+ Android 小组件 (差异化，接近原版)
  ↓ 或 W13 (选项 B)
+ GitHub 同步 (Ticking 做不了的)
```

**选项 A**: 重视体验完整度（贴近原版 Ticking）
- Beta 中加小组件
- 缺点: 时间长

**选项 B**: 重视创新度（超越原版 Ticking）
- Beta 不做小组件，做 GitHub 同步
- 优势: 本地 + 云端结合，用户不怕数据丢失
- 缺点: 实现复杂

## 7. MVP 定义（最关键）

### "真正的 MVP" 是什么？

**错误定义**: "能启动 App"
**正确定义**: "用户按照预期用起来，能形成一个完整的工作流"

对于 Ticking:

```
最小工作流:
┌─────────────────────────────────┐
│ 1. 用户打开 App                  │
│ 2. 点"开始计时" (25 min 番茄)    │
│ 3. 计时运行，显示倒计时          │
│ 4. 计时完成，收到通知            │
│ 5. 自动记录这次计时              │
│ 6. 用户可以查看历史记录          │
└─────────────────────────────────┘
```

**这个流程需要的功能**:
- 计时器页面 ✅
- 7 种计时模式 ✅
- 本地数据库 ✅
- 通知 ✅
- 简单的历史列表 ✅
- **不需要**：热力图、待办、小组件、GitHub 同步

**这是 v0.3 的范围** = **真正的 MVP**

## 8. 最终建议

### 你应该这样做：

**✅ DO**:
1. **W1-W4 闭门造车** — 只给自己用，验证"计时"是否真的有价值
2. **W5-W6 快速发布** — GitHub Release v0.4（待办）& v1.0.0（热力图）
3. **W7-W8 收集反馈** — 邀请 10-20 个开源爱好者试用
4. **W9-W10 根据反馈** — 用户要小组件就做，要 GitHub 同步就做
5. **W11-W13 打磨上架** — 性能优化 + 应用商店发布

**❌ DON'T**:
- ❌ 一开始就把所有功能做完（浪费时间）
- ❌ 等 v1.0 完全完美才发布（永远发不了）
- ❌ 忽视用户反馈（白做功能）
- ❌ 太多次大改架构（浪费精力）

**📊 成功指标**:
- W8 末: 有 50+ GitHub Star（技术社区认可）
- W13 末: 有 1000+ 日活用户（真实反馈）
- 用户坚持率 > 30%（一个月后还在用）
